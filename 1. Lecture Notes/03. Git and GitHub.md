# Git and GitHub

---

## What are Git and GitHub?

**Git** is a **version control system** — a tool that tracks changes to your files over time. It lets you save snapshots of your project at any point, see what changed, undo mistakes, and collaborate with others without overwriting each other's work.

**GitHub** is a website that hosts Git repositories in the cloud. Git is the tool; GitHub is the platform. You can use Git entirely on your own computer without GitHub, but GitHub makes it easy to share code, collaborate, and back up your work online.

| | Git | GitHub |
|---|---|---|
| **What it is** | Software installed on your computer | A website / cloud service |
| **What it does** | Tracks file history locally | Hosts repositories online |
| **Works without the other?** | Yes | No — GitHub needs Git |

---

## Core Concepts

### Repository (repo)
A **repository** is a folder that Git is tracking. It contains all your project files plus a hidden `.git` folder where Git stores the entire history of changes.

### Commit
A **commit** is a saved snapshot of your project at a specific point in time. Think of it like a checkpoint in a video game — you can always return to any previous commit. Each commit has:
- A unique ID (called a **hash**, e.g. `a3f9c12`)
- A **commit message** describing what changed
- A timestamp and author

### Staging Area (Index)
Before you commit, you have to **stage** your changes — this is a holding area where you choose exactly which changes to include in the next commit. This lets you commit only some of your changes if you have multiple unrelated edits.

```
Working Directory  →  Staging Area  →  Repository (committed history)
   (edit files)       (git add)           (git commit)
```

### Branch
A **branch** is an independent line of development. The default branch is usually called **main** (or **master** in older projects). You create branches to work on new features or fixes without affecting the main codebase, then **merge** them back when ready.

```
main:    A --- B --- C
                      \
feature:               D --- E
```

### Remote
A **remote** is a version of your repository hosted somewhere else (e.g. on GitHub). The default remote is conventionally named **origin**.

## The Git Workflow

The typical day-to-day workflow looks like this:

1. **Make changes** to your files in your working directory
2. **Stage** the changes you want to save with `git add`
3. **Commit** the staged changes with `git commit`
4. **Push** your commits to GitHub with `git push`

## Essential Commands

### Setup (one time)
```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```

### Starting a repository
```bash
git init                    # Turn the current folder into a Git repo
git clone <url>             # Download an existing repo from GitHub
```

### Checking status
```bash
git status                  # See what files have changed or are staged
git log                     # See the commit history
git log --oneline           # Compact version of the commit history
git diff                    # See unstaged changes line by line
```

### Staging and committing
```bash
git add <file>              # Stage a specific file
git add .                   # Stage all changed files in the current folder
git commit -m "message"     # Commit staged changes with a message
```

### Working with GitHub (remotes)
```bash
git remote add origin <url> # Link your local repo to a GitHub repo
git push origin main        # Upload your commits to GitHub
git pull origin main        # Download and merge changes from GitHub
git fetch origin            # Download changes from GitHub without merging
```

### Branching
```bash
git branch                  # List all branches
git branch <name>           # Create a new branch
git switch <name>           # Switch to a branch
git switch -c <name>        # Create and switch to a new branch in one step
git merge <name>            # Merge a branch into the current branch
git branch -d <name>        # Delete a branch (safe — won't delete if unmerged)
```

## Key Terms Glossary

| Term | Definition |
|------|-----------|
| **Repository (repo)** | A project folder tracked by Git |
| **Commit** | A saved snapshot of your project |
| **Stage / Index** | The holding area for changes before committing |
| **Branch** | An independent line of development |
| **Merge** | Combining changes from one branch into another |
| **Merge conflict** | When two branches changed the same line differently — Git needs you to resolve it manually |
| **Remote** | A copy of the repo hosted elsewhere (e.g. GitHub) |
| **Origin** | The conventional name for the default remote |
| **Push** | Upload local commits to the remote |
| **Pull** | Download remote commits and merge them locally |
| **Fetch** | Download remote commits without merging |
| **Clone** | Copy a remote repo to your local machine |
| **Fork** | Your personal copy of someone else's GitHub repo |
| **Pull Request (PR)** | A GitHub feature to propose merging changes, with discussion and review |
| **HEAD** | A pointer to the commit you're currently on (usually the tip of your branch) |
| **Checkout** | Switching to a different branch or commit (older command; `switch` is preferred now) |
| **.gitignore** | A file listing paths Git should not track (e.g. `node_modules/`, `.env`) |

## Undoing Things

```bash
git restore <file>          # Discard unstaged changes to a file (can't undo!)
git restore --staged <file> # Unstage a file (moves it back to working directory)
git revert <hash>           # Create a new commit that undoes a past commit (safe)
git reset --hard <hash>     # Move HEAD back to a commit, discarding all later changes (destructive!)
```

> **Rule of thumb:** prefer `git revert` over `git reset --hard` when working with shared branches, since revert preserves history and doesn't rewrite commits others may have.

## A Typical GitHub Flow

This is a common workflow for contributing to a project:

1. **Fork** the repo on GitHub (creates your personal copy)
2. **Clone** your fork to your machine: `git clone <your-fork-url>`
3. Create a **feature branch**: `git switch -c my-feature`
4. Make your changes, then **stage and commit** them
5. **Push** your branch to your fork: `git push origin my-feature`
6. Open a **Pull Request** on GitHub from your branch to the original repo's `main`
7. After review and approval, the PR is **merged**

## Good Commit Message Habits

A commit message should complete the sentence: *"If applied, this commit will..."*

```
✅ Add navigation bar to homepage
✅ Fix typo in login error message
✅ Remove unused CSS variables

❌ stuff
❌ fix
❌ changes
```

Keep the first line under 72 characters. If more detail is needed, leave a blank line and add a longer description below.

## .gitignore

A `.gitignore` file tells Git which files to ignore entirely — they won't be tracked, staged, or committed.

```gitignore
# Dependencies
node_modules/

# Environment variables (never commit secrets!)
.env

# Build output
dist/
build/

# OS files
.DS_Store
Thumbs.db
```

Create this file at the root of your repo. GitHub provides templates for most languages and frameworks.
