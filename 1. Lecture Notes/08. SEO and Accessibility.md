# SEO and Accessibility

---

## What Is SEO?

SEO (Search Engine Optimisation) is the practice of making your website appear higher in search results on Google and other search engines. When someone searches for something relevant to your site, you want to appear as high as possible — the difference between rank #1 and rank #5 is enormous in click-through rate.

Search engines like Google send **bots** (also called crawlers or spiders) to read your website. They analyse your content, structure, and many other signals to decide how relevant you are for given search terms.

## What Is Accessibility?

Accessibility (often abbreviated **a11y** — 11 letters between the 'a' and 'y') means making your website usable by as many people as possible, including people with:
- Visual impairments (blindness, low vision, colour blindness)
- Motor impairments (difficulty using a mouse — keyboard-only users)
- Cognitive impairments (dyslexia, ADHD)
- Hearing impairments (for audio/video content)

Accessibility and SEO are deeply connected. Search engines are essentially very sophisticated screen readers — they can only "see" your content if it's well-structured and semantic.

## HTML Structure for SEO

The most important SEO factor is **content quality and relevance**. But good HTML structure helps search engines understand what your page is about.

### Use semantic HTML

Search engines give more weight to content inside semantic tags:

```html
<article>
  <header>
    <h1>How to Bake Sourdough Bread</h1>  <!-- Main topic of the page -->
    <time datetime="2025-01-15">January 15, 2025</time>
  </header>

  <p>The introduction paragraph signals to Google what the article covers.</p>

  <h2>Ingredients</h2>           <!-- Subtopic — h2 weighted less than h1 -->
  <h3>For the starter</h3>       <!-- Sub-subtopic -->
  ...
</article>
```

**Rules for headings:**
- One `<h1>` per page — it's the main topic
- Don't skip levels (don't go from `h1` to `h4`)
- Make headings descriptive — they appear in search snippets

### URL structure

URLs should be readable by humans — they signal content to search engines:

```
❌ Bad:  https://example.com/p?id=4829
✓ Good: https://example.com/blog/how-to-bake-sourdough-bread
```

- Use hyphens to separate words (not underscores)
- Keep URLs short and descriptive
- Avoid unnecessary words ("the", "a", "in")

## Meta Tags

Meta tags go inside `<head>` and provide information about the page to search engines and social media platforms.

```html
<head>
  <!-- Title tag: appears in browser tab and Google search results -->
  <!-- Aim for 50-60 characters. Make it unique per page. Include target keyword. -->
  <title>How to Bake Sourdough Bread — Complete Beginner's Guide | BakingBlog</title>

  <!-- Meta description: the text snippet shown under the title in search results -->
  <!-- Not a ranking factor, but affects click-through rate. 150-160 characters max. -->
  <meta name="description" content="Learn to bake sourdough bread from scratch with this step-by-step guide. Covers starter, hydration, shaping, and baking. Perfect for beginners.">

  <!-- Canonical: tells Google this is the "real" URL if the same content is at multiple URLs -->
  <link rel="canonical" href="https://example.com/blog/how-to-bake-sourdough-bread">

  <!-- Robots: control what search engines can index -->
  <meta name="robots" content="index, follow">     <!-- default — index this page -->
  <meta name="robots" content="noindex, nofollow"> <!-- don't index, don't follow links -->
</head>
```

### Open Graph (social media previews)

When someone shares your link on Twitter, Facebook, Slack, or LinkedIn, these tags control what the preview card looks like:

```html
<!-- Open Graph — used by Facebook, LinkedIn, most platforms -->
<meta property="og:title" content="How to Bake Sourdough Bread">
<meta property="og:description" content="A complete beginner's guide to sourdough.">
<meta property="og:image" content="https://example.com/images/sourdough-og.jpg">
<!-- Image should be 1200×630px for best results -->
<meta property="og:url" content="https://example.com/blog/how-to-bake-sourdough-bread">
<meta property="og:type" content="article">

<!-- Twitter-specific -->
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="How to Bake Sourdough Bread">
<meta name="twitter:description" content="A complete beginner's guide to sourdough.">
<meta name="twitter:image" content="https://example.com/images/sourdough-twitter.jpg">
```

## Structured Data (Schema.org)

Structured data is additional machine-readable information you embed in your HTML using JSON-LD format. It helps Google show **rich results** in search — star ratings, FAQ accordions, recipe cards, event dates, etc.

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Recipe",
  "name": "Sourdough Bread",
  "author": {
    "@type": "Person",
    "name": "Alice Baker"
  },
  "datePublished": "2025-01-15",
  "description": "A simple sourdough bread recipe for beginners.",
  "totalTime": "PT24H",
  "recipeYield": "1 loaf",
  "recipeIngredient": [
    "500g bread flour",
    "375ml water",
    "10g salt"
  ],
  "aggregateRating": {
    "@type": "AggregateRating",
    "ratingValue": "4.8",
    "reviewCount": "127"
  }
}
</script>
```

Google's [Rich Results Test](https://search.google.com/test/rich-results) validates your structured data.

## Technical SEO

### Sitemap

A sitemap is an XML file listing all your pages. It helps search engines discover and crawl your content, especially for large sites.

```xml
<!-- public/sitemap.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://example.com/</loc>
    <lastmod>2025-01-15</lastmod>
    <priority>1.0</priority>
  </url>
  <url>
    <loc>https://example.com/blog</loc>
    <changefreq>daily</changefreq>
    <priority>0.8</priority>
  </url>
</urlset>
```

Submit your sitemap in Google Search Console.

### robots.txt

Tells search engine bots what to crawl and what to skip:

```
# public/robots.txt
User-agent: *
Allow: /
Disallow: /admin/
Disallow: /private/
Sitemap: https://example.com/sitemap.xml
```

### Core Web Vitals

Google uses these performance metrics as ranking factors:

| Metric | What it measures | Good threshold |
| ------ | ---------------- | -------------- |
| **LCP** (Largest Contentful Paint) | How fast the main content loads | < 2.5 seconds |
| **FID** / **INP** (Interaction to Next Paint) | How fast the page responds to input | < 200ms |
| **CLS** (Cumulative Layout Shift) | How much the layout jumps around while loading | < 0.1 |

Measure with: PageSpeed Insights, Lighthouse (Chrome DevTools → Lighthouse tab), or Google Search Console.

## Accessibility Fundamentals

### WCAG Guidelines

The **Web Content Accessibility Guidelines (WCAG)** are the international standard for web accessibility. The four principles (POUR):

| Principle | Meaning |
| --------- | ------- |
| **Perceivable** | Users can perceive all content (visible, audible, or via screen reader) |
| **Operable** | Users can navigate and interact using any input method |
| **Understandable** | Content and UI are clear and predictable |
| **Robust** | Works with current and future assistive technologies |

Levels: A (minimum), AA (standard target), AAA (maximum — not always achievable).

---

## Screen Readers

Screen readers convert on-screen content to speech or braille output. Popular ones: NVDA (Windows, free), JAWS (Windows, paid), VoiceOver (Mac/iOS, built-in), TalkBack (Android).

Screen readers navigate by:
- Headings (jumping between `h1`, `h2`, etc.)
- Landmarks (nav, main, footer regions)
- Links and buttons
- Form labels

This is why semantic HTML is so critical — a `<div>` clicked like a button is invisible to a screen reader.

### How to make interactive elements accessible

```html
<!-- Bad: div pretending to be a button -->
<!-- Screen reader can't focus it, no keyboard support -->
<div onclick="submit()">Submit</div>

<!-- Good: real button -->
<!-- Focusable, fires on Enter/Space, announced as "Submit, button" -->
<button type="button" onclick="submit()">Submit</button>

<!-- Icon-only button: must have an accessible name -->
<button aria-label="Close dialog">
  <svg aria-hidden="true">...</svg>   <!-- hide decorative icon from screen reader -->
</button>

<!-- Link with descriptive text -->
<!-- Bad: "click here" says nothing about destination -->
<a href="/report.pdf">Click here</a>

<!-- Good: destination is clear -->
<a href="/report.pdf">Download annual report (PDF, 2MB)</a>
```

## ARIA

ARIA (Accessible Rich Internet Applications) attributes add semantic meaning that HTML alone can't express — particularly for complex interactive widgets.

**The first rule of ARIA**: don't use ARIA if a native HTML element can do it. `<button>` is always better than `<div role="button">`.

```html
<!-- role: tells screen readers what kind of element this is -->
<div role="alert" aria-live="polite">
  Your form was saved successfully.
</div>

<!-- aria-label: provides a text label when visual label isn't enough -->
<input type="search" aria-label="Search products">

<!-- aria-labelledby: points to an element that labels this one -->
<h2 id="section-heading">Recent Orders</h2>
<table aria-labelledby="section-heading">...</table>

<!-- aria-describedby: points to extra descriptive text -->
<input type="password" aria-describedby="pw-hint">
<p id="pw-hint">Must be at least 8 characters with one number.</p>

<!-- aria-expanded: for toggleable elements -->
<button aria-expanded="false" aria-controls="dropdown-menu">
  Settings
</button>
<ul id="dropdown-menu" hidden>...</ul>

<!-- aria-hidden: hide decorative content from screen readers -->
<span aria-hidden="true">→</span>

<!-- aria-live: announce dynamic content changes -->
<div aria-live="polite">   <!-- reads changes when user is idle -->
<div aria-live="assertive"> <!-- interrupts to read changes immediately -->
```

## Keyboard Navigation

Every user action must be achievable with only a keyboard. This is required by WCAG and is also how power users navigate.

```css
/* Never hide focus outlines entirely — keyboard users need them */
/* Bad: */
:focus { outline: none; }

/* Good: a visible, attractive focus style */
:focus-visible {
  outline: 3px solid #3b82f6;
  outline-offset: 2px;
  border-radius: 4px;
}

/* Use :focus-visible (not :focus) to only show outline for keyboard navigation,
   not for mouse clicks */
```

```html
<!-- tabindex controls keyboard focus order -->
<!-- tabindex="0": element can be focused (normal tab order) -->
<div tabindex="0" role="button" onkeydown="handleKey(event)">Custom Button</div>

<!-- tabindex="-1": focusable programmatically but not in tab order -->
<!-- Use for modals: move focus inside on open, back on close -->
<div tabindex="-1" id="modal">...</div>

<!-- Never use tabindex > 0 — it breaks the natural tab order -->
```

## Colour Contrast

Text must have sufficient contrast against its background so people with low vision or colour blindness can read it.

**WCAG AA requirements:**
- Normal text (< 18pt): contrast ratio of **4.5:1**
- Large text (≥ 18pt or 14pt bold): contrast ratio of **3:1**
- Non-text UI elements (borders, icons): **3:1**

Tools:
- WebAIM Contrast Checker — paste two colours, get the ratio
- Colour & Contrast (browser extension)
- Figma plugins: Contrast, A11y — Colour Contrast Checker

```css
/* Good contrast: white on #3b82f6 (blue) → 3.07:1 — only passes for large text */
/* Better: white on #1d4ed8 (darker blue) → 5.91:1 — passes for all text */
.button {
  background: #1d4ed8;
  color: white;
}
```

## Reduced Motion

Some users experience nausea, dizziness, or seizures from animations and motion effects. Always respect the operating system's reduced motion preference:

```css
/* By default: animations are on */
.card {
  transition: transform 0.3s ease;
}

.card:hover {
  transform: scale(1.05);
}

/* If the OS has "Reduce Motion" enabled, disable animations */
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

/* Or: provide an alternative for motion-sensitive users */
@media (prefers-reduced-motion: no-preference) {
  .hero {
    animation: fadeSlideIn 0.5s ease forwards;
  }
}
```

In JavaScript:

```js
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;

if (!prefersReducedMotion) {
  startAnimation();
}
```

## Dark Mode

Respect the user's OS colour scheme preference:

```css
:root {
  --color-bg: white;
  --color-text: #1e293b;
  --color-primary: #3b82f6;
}

@media (prefers-color-scheme: dark) {
  :root {
    --color-bg: #0f172a;
    --color-text: #e2e8f0;
    --color-primary: #60a5fa;
  }
}

body {
  background: var(--color-bg);
  color: var(--color-text);
}
```

## Images

```html
<!-- Informative image: alt text describes the image content -->
<img src="sourdough.jpg" alt="A golden-brown sourdough loaf cooling on a wire rack">

<!-- Decorative image: empty alt hides it from screen readers -->
<!-- (The img is purely visual — a screen reader skipping it doesn't lose information) -->
<img src="divider.png" alt="">

<!-- Complex image (charts, diagrams): describe in text as well -->
<figure>
  <img src="chart.png" alt="Bar chart showing monthly revenue">
  <figcaption>
    Revenue grew from £50k in January to £120k in December, with the largest jump in Q4.
  </figcaption>
</figure>
```

## Forms

Forms are the most common accessibility failure on the web.

```html
<!-- Always pair labels with inputs using for/id -->
<label for="email">Email address</label>
<input type="email" id="email" name="email" required autocomplete="email">

<!-- Required field indicator -->
<label for="name">
  Full name <span aria-hidden="true">*</span>
  <span class="sr-only">(required)</span>
</label>

<!-- Error messages linked to the input -->
<input
  type="email"
  id="email"
  aria-describedby="email-error"
  aria-invalid="true"
>
<p id="email-error" role="alert">Please enter a valid email address.</p>

<!-- .sr-only: visible to screen readers but hidden visually -->
<style>
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
</style>
```

## Testing Accessibility

| Tool | How to use | What it finds |
| ---- | ---------- | ------------- |
| **Lighthouse** | Chrome DevTools → Lighthouse → Accessibility | Automated issues, score |
| **axe DevTools** | Browser extension | Detailed automated audit |
| **WAVE** | Browser extension | Visual overlay of issues |
| **Keyboard test** | Unplug your mouse | Can you do everything with Tab/Enter/Space/Arrow keys? |
| **Screen reader test** | Mac: `Cmd+F5` for VoiceOver | Does the page make sense without seeing it? |
| **Colour contrast checker** | WebAIM or Figma plugin | Contrast ratios |
| **Browser zoom** | `Ctrl+` up to 200% | Does content reflow correctly? |
