# TypeScript Foundations

---

## Why Learn TypeScript When React Uses JavaScript?

React itself is written in JavaScript and runs on JavaScript — you can absolutely build React apps without ever touching TypeScript. So why bother?

The short answer: **JavaScript is fine for small projects; TypeScript is necessary for large ones.**

When a React codebase grows past a few files, JavaScript's lack of types becomes a liability:

- You rename a prop in one component and silently break three others — no error until a user hits the bug
- You call `user.profie.name` (typo) and it crashes at runtime, not while you're coding
- A new developer joins and has no idea what shape of data a component expects — they have to read every usage

TypeScript solves all of this. Your editor knows the exact shape of every prop, every API response, every function argument. Mistakes become red underlines in your editor instead of crashes in production.

**The tradeoff**: TypeScript requires an extra compilation step, and writing types adds some upfront work. For a weekend project, plain JavaScript is fine. For anything you'll maintain for months or ship to real users, TypeScript is the professional standard — virtually every serious React codebase at companies uses it.

```tsx
// Without TypeScript — no idea what props are required
function UserCard({ user }) {
  return <div>{user.naem}</div>; // typo — crashes silently
}

// With TypeScript — error caught immediately
interface User {
  id: number;
  name: string;
  email: string;
}

function UserCard({ user }: { user: User }) {
  return <div>{user.naem}</div>; // TS ERROR: Property 'naem' does not exist
}
```

## What Is TypeScript?

TypeScript is a **superset of JavaScript** developed by Microsoft. "Superset" means every single JavaScript file is also valid TypeScript — you don't throw your code away, you gradually add types to it. TypeScript is compiled down to plain JavaScript before it runs; the browser or Node.js never sees TypeScript directly.

The core problem TypeScript solves: JavaScript lets you do anything, which is both its strength and its weakness. You can call `user.naem` when you meant `user.name` and JavaScript won't tell you until the code actually runs and crashes — which might be in production, seen by real users. TypeScript tells you about that typo the instant you type it, because it knows the shape of `user` and can see that `naem` doesn't exist on it.

Think of TypeScript as a spell-checker for your code's logic, not just its spelling.

| | JavaScript | TypeScript |
| - | ---------- | ---------- |
| Bugs found | When code runs (may be in production) | When you type, in your editor |
| Autocomplete | Limited (editor guesses) | Rich (editor knows exact shape of every value) |
| Refactoring | Dangerous — easy to miss a usage | Safe — rename a property and every usage is updated |
| File extension | `.js` | `.ts` (or `.tsx` for JSX/React) |

## Setting Up TypeScript

```bash
# Install TypeScript as a dev dependency
npm install -D typescript

# Create a tsconfig.json with sensible defaults
npx tsc --init
```

### tsconfig.json

This file tells TypeScript how to compile your code. The most important options:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    // What version of JavaScript to output. ES2022 supports modern features
    // and works in all current browsers and Node.js versions.

    "module": "ESNext",
    // Which module system to use. ESNext gives you import/export syntax.

    "strict": true,
    // THE most important option. Enables all strict type checks as a bundle.
    // Always turn this on — without it, TypeScript is much weaker.

    "outDir": "./dist",
    // Where the compiled JavaScript files go.

    "rootDir": "./src",
    // Where your TypeScript source files live.

    "sourceMap": true
    // Creates .js.map files that let browser DevTools show you the original
    // TypeScript instead of the compiled JavaScript when debugging.
  }
}
```

To compile: run `npx tsc` (one-time) or `npx tsc --watch` (auto-recompile on every save).

## Type Annotations: Telling TypeScript What Something Is

In JavaScript, a variable can hold anything. In TypeScript, you can specify (or let TypeScript infer) what type a variable holds. Once it has a type, TypeScript stops you from misusing it.

```ts
// The syntax: variableName: type = value
let name: string = 'Alice';
let age: number = 28;
let active: boolean = true;
let nothing: null = null;
let missing: undefined = undefined;

// Now TypeScript knows what these are — and will stop you from doing silly things:
name.toUpperCase();  // fine — strings have toUpperCase
name + 10;           // TypeScript warns: you're adding a string and a number. Intentional?
age.length;          // Error: numbers don't have a length property
```

### The basic types

```ts
// string: text
let username: string = 'alice_99';
let message: string = `Hello, ${username}!`;  // template literals work too

// number: ALL numbers (integers and decimals — no separate int/float)
let score: number = 95;
let price: number = 19.99;
let negative: number = -7;

// boolean: true or false
let isLoggedIn: boolean = true;
let hasPermission: boolean = false;

// null and undefined
// null: deliberately empty
// undefined: not yet assigned
let user: null = null;
let result: undefined = undefined;

// Arrays: specify the type of elements inside []
let scores: number[] = [90, 85, 92];   // array of numbers
let names: string[] = ['Alice', 'Bob']; // array of strings
let flags: boolean[] = [true, false, true];
let items: Array<string> = ['a', 'b']; // alternative syntax — same thing

// void: a function that doesn't return anything
function log(message: string): void {
  console.log(message);
  // no return statement — returns undefined implicitly
}

// never: a function that NEVER completes (throws an error or loops forever)
function crash(message: string): never {
  throw new Error(message);
  // TypeScript knows code after this is unreachable
}
```

## Type Inference: TypeScript Is Smart

You don't always need to write the type — TypeScript can often figure it out from the value you assign. This is called **type inference**, and it's one of TypeScript's most powerful features.

```ts
// TypeScript infers these types from the right-hand side value:
const name = 'Alice';       // inferred: string
const age = 28;             // inferred: number
const active = true;        // inferred: boolean
const scores = [1, 2, 3];  // inferred: number[]

// These are completely equivalent:
const name: string = 'Alice';
const name = 'Alice';           // TypeScript already knows it's a string

// In VS Code: hover over any variable to see what TypeScript has inferred.
// This is one of the most useful habits to build.
```

**When to write explicit type annotations:**

- Function parameters — TypeScript cannot infer these from usage alone
- When you want a wider type than what would be inferred (e.g., `string | null`)
- For the return type of functions, when you want to guarantee it explicitly

```ts
// TypeScript CAN'T infer parameter types — you must annotate them:
function add(a: number, b: number): number {
  return a + b;
}

// Return type annotation is optional here (TypeScript can infer it's number),
// but writing it explicitly is a good habit — it documents your intent.
```

## Interfaces: Describing the Shape of Objects

An **interface** is like a contract that describes what properties an object must have and what types those properties must be.

```ts
interface User {
  id: number;
  name: string;
  email: string;
  role?: string;        // the ? makes this property optional — it can be omitted
  readonly createdAt: Date;  // readonly: can be set once, never changed after
}

// Any object that has these properties (and the right types) satisfies this interface
const alice: User = {
  id: 1,
  name: 'Alice',
  email: 'alice@example.com',
  createdAt: new Date(),
  // role is omitted — that's fine because it's optional
};

// TypeScript will error if:
// - you're missing a required property
// - a property has the wrong type
// - you try to assign to readonly

alice.name = 'Alice Smith'; // fine — not readonly
alice.createdAt = new Date(); // Error: createdAt is readonly
```

### Why interfaces are useful

Without an interface, TypeScript just treats an object as "some object". With an interface, TypeScript knows exactly what it contains, giving you autocomplete and error checking on every property access.

```ts
function greetUser(user: User) {
  // TypeScript knows user has .name (string), so this is fine:
  return `Hello, ${user.name}!`;

  // TypeScript would catch this typo immediately:
  return `Hello, ${user.naem}!`; // Error: Property 'naem' does not exist on type 'User'
}
```

## Type Aliases

A **type alias** is another way to name a type. Use `type` for unions, primitive aliases, and function types. Use `interface` for object shapes (either works, but this is the convention):

```ts
type Status = 'active' | 'inactive' | 'suspended';  // a union — see next section
type UserID = string | number;
type Callback = (error: Error | null, result: string) => void;

type User = {
  id: number;
  name: string;
};
```

## Union Types: One OR Another

A **union type** says "this value can be one of these types." Use the pipe `|` to combine them.

```ts
// A function parameter that accepts either a string or a number
function formatId(id: string | number): string {
  return id.toString();
}

formatId(42);       // fine
formatId('abc-99'); // fine
formatId(true);     // Error — boolean is not in the union

// Literal union types: specific values instead of general types
type Status = 'loading' | 'success' | 'error';
type Direction = 'north' | 'south' | 'east' | 'west';

let status: Status = 'loading'; // fine
status = 'pending';              // Error — 'pending' is not a valid Status

// Optional properties are really "value or undefined" under the hood:
// role?: string   is the same as   role: string | undefined
```

## Functions

TypeScript brings type safety to function parameters and return values — the most common source of bugs in JavaScript.

```ts
// Annotate parameters and return type
function add(a: number, b: number): number {
  return a + b;
}

// Optional parameters: add ? after the name
// Optional parameters must come after required ones
function greet(name: string, greeting?: string): string {
  return `${greeting ?? 'Hello'}, ${name}!`;
}

greet('Alice');           // 'Hello, Alice!'
greet('Alice', 'Hi');     // 'Hi, Alice!'

// Default parameter values
function createUser(name: string, role: string = 'viewer'): User {
  return { name, role };
}

// Rest parameters: collect remaining arguments into an array
function sum(first: number, ...rest: number[]): number {
  return rest.reduce((total, n) => total + n, first);
}

sum(1, 2, 3, 4); // 10

// Function type: describe what a function looks like (useful for callbacks)
type Transformer = (input: string) => string;

function applyTransform(text: string, fn: Transformer): string {
  return fn(text);
}

applyTransform('  hello  ', str => str.trim()); // 'hello'
```

## Enums: Named Constants

An **enum** (enumeration) is a set of named constants. They prevent magic strings or numbers from littering your code.

```ts
// Without enum: you'd write 'PENDING', 'APPROVED', 'REJECTED' as strings everywhere
// If you make a typo, TypeScript won't catch it unless you use a type alias

// String enum: the values are meaningful strings
enum OrderStatus {
  Pending = 'PENDING',
  Approved = 'APPROVED',
  Rejected = 'REJECTED',
  Shipped = 'SHIPPED',
}

function processOrder(status: OrderStatus) {
  if (status === OrderStatus.Approved) {
    shipOrder();
  }
}

processOrder(OrderStatus.Approved); // fine
processOrder('APPROVED');            // Error — must use the enum

// Numeric enum (simpler, but values are less readable)
enum Direction {
  Up,    // 0
  Down,  // 1
  Left,  // 2
  Right, // 3
}
```

**When to use a union type instead of an enum:** For most cases, a literal union type is simpler and doesn't add any runtime code:

```ts
type OrderStatus = 'PENDING' | 'APPROVED' | 'REJECTED' | 'SHIPPED';
// Simpler — no import needed, no runtime overhead
```

Use enums when you need to iterate over the values or when the numeric values matter.

## Classes in TypeScript

TypeScript adds access modifiers to JavaScript classes, which control where a property or method can be used:

- `public` — accessible anywhere (the default if you don't write anything)
- `private` — only accessible within this class
- `protected` — accessible within this class and any subclass

```ts
class BankAccount {
  private balance: number;   // only this class can read or change balance
  public owner: string;
  readonly id: string;       // can be set in the constructor, never again

  constructor(owner: string, initialBalance: number) {
    this.owner = owner;
    this.balance = initialBalance;
    this.id = Math.random().toString(36);
  }

  deposit(amount: number): void {
    if (amount <= 0) throw new Error('Deposit amount must be positive');
    this.balance += amount;
  }

  withdraw(amount: number): void {
    if (amount > this.balance) throw new Error('Insufficient funds');
    this.balance -= amount;
  }

  // Getter: accessed like a property (account.balance) but computed
  get currentBalance(): number {
    return this.balance;
  }
}

const account = new BankAccount('Alice', 1000);
account.deposit(500);
account.currentBalance;  // 1500
account.balance;          // Error! balance is private
account.id = 'new-id';   // Error! id is readonly
```

### Shorthand constructor parameter properties

TypeScript lets you declare and assign class properties in one step:

```ts
// Verbose version:
class Person {
  public name: string;
  private age: number;

  constructor(name: string, age: number) {
    this.name = name;
    this.age = age;
  }
}

// Shorthand: same result with less code
class Person {
  constructor(
    public name: string,
    private age: number,
  ) {} // TypeScript creates the properties and assigns them automatically
}
```

## Type Narrowing: Working with Multiple Types

When a variable could be one of several types, TypeScript **narrows** down which type it is based on the checks you write. Inside the narrowed block, TypeScript knows the specific type and gives you the right autocomplete and error checking.

```ts
function describe(value: string | number) {
  // Here, value is string | number — TypeScript doesn't know which

  if (typeof value === 'string') {
    // Inside this block, TypeScript has narrowed: value is definitely string
    console.log(value.toUpperCase()); // fine — strings have toUpperCase
    console.log(value.toFixed(2));    // Error! toFixed is for numbers
  } else {
    // Outside the if: TypeScript knows it must be number
    console.log(value.toFixed(2));    // fine
  }
}

// Narrowing with instanceof (for classes and built-ins like Date)
function formatDate(value: Date | string): string {
  if (value instanceof Date) {
    return value.toISOString(); // value is Date here
  }
  return value; // value is string here
}

// Narrowing with "in" (check if a property exists)
interface Cat { meow(): void; }
interface Dog { bark(): void; }

function makeNoise(pet: Cat | Dog): void {
  if ('meow' in pet) {
    pet.meow(); // TypeScript knows it's a Cat here
  } else {
    pet.bark(); // TypeScript knows it's a Dog here
  }
}
```

Narrowing is why TypeScript is safe while still being flexible — you can accept multiple types and handle each one correctly, and TypeScript verifies you've done so.

## `any` and `unknown`: The Escape Hatches

Sometimes you genuinely don't know a type — like data coming from an API, user input, or old JavaScript code. TypeScript provides two types for this:

### `any` — disables type checking entirely

```ts
let data: any = fetchSomething();
data.anything;          // TypeScript lets this through — no checking at all
data();                 // TypeScript allows this even if data isn't a function
data.foo.bar.baz;       // TypeScript has no idea if this works

// any spreads — if you assign any to another variable, that becomes any too
const name: string = data.name;  // TypeScript thinks this is fine... but is it really?
```

`any` is a last resort. It defeats the purpose of TypeScript. Avoid it except when migrating old code.

### `unknown` — the safe alternative

```ts
let data: unknown = fetchSomething();
data.anything;    // Error! You can't access properties on unknown without checking first
data();           // Error! You can't call unknown without checking first

// You must narrow it before using it:
if (typeof data === 'string') {
  data.toUpperCase(); // fine — narrowed to string
}

if (typeof data === 'object' && data !== null && 'name' in data) {
  console.log(data.name); // fine — narrowed
}
```

Use `unknown` whenever you'd be tempted to use `any` — it forces you to verify the type before using it, which is exactly what you should do with untrusted data.
