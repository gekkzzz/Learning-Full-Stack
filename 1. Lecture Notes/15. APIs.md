# APIs

---

## What Is an API?

An **API** (Application Programming Interface) is a defined way for two pieces of software to communicate with each other. The word "interface" is the key idea — it's a contract that says "if you ask me a question in this format, I'll give you an answer in this format."

In web development, "API" almost always means a **Web API** — a server on the internet that you communicate with over HTTP. You send a request; the server sends back data (almost always JSON).

A real-world analogy: a restaurant menu is an API. It tells you what you can order (the available endpoints), how to order (the format), and what you'll get back (the response). The kitchen (the server) handles all the complexity behind the scenes — you don't need to know how the food is made.

---

## REST — The Standard

**REST** (Representational State Transfer) is the architectural style most Web APIs follow. A REST API is organised around **resources** (things: users, products, orders) and uses HTTP **methods** to express what you want to do with them.

### HTTP Methods

| Method | Meaning | Example |
| ------ | ------- | ------- |
| `GET` | Read a resource | Get a user's details |
| `POST` | Create a new resource | Create a new user |
| `PUT` | Replace a resource entirely | Replace all user data |
| `PATCH` | Update part of a resource | Change just the user's email |
| `DELETE` | Delete a resource | Delete a user |

### URL structure (endpoints)

Resources are identified by URLs. The URL should describe the resource, not the action — the HTTP method expresses the action.

```
GET    /users           → list all users
POST   /users           → create a new user
GET    /users/42        → get user with ID 42
PUT    /users/42        → replace user 42
PATCH  /users/42        → partially update user 42
DELETE /users/42        → delete user 42

GET    /users/42/posts  → get all posts by user 42
POST   /users/42/posts  → create a new post for user 42
```

## HTTP in Detail

Every API interaction is an HTTP request followed by an HTTP response. Understanding the structure helps you debug API problems.

### Request structure

```
POST /users HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbG...
Content-Type: application/json
Accept: application/json

{
  "name": "Alice",
  "email": "alice@example.com"
}
```

| Part | What it is |
| ---- | ---------- |
| Method + path | What you want to do and where |
| Host | The server's domain |
| Headers | Metadata: authentication token, content type, etc. |
| Body | The data you're sending (for POST/PUT/PATCH) |

### Response structure

```
HTTP/1.1 201 Created
Content-Type: application/json
Location: /users/42

{
  "id": 42,
  "name": "Alice",
  "email": "alice@example.com",
  "createdAt": "2025-01-15T10:23:00Z"
}
```

### Status codes

Status codes tell you what happened. The first digit is the category:

| Range | Category | Meaning |
| ----- | -------- | ------- |
| 2xx | Success | The request worked |
| 3xx | Redirect | The resource has moved |
| 4xx | Client error | **You** made a mistake |
| 5xx | Server error | The **server** made a mistake |

Common codes:

| Code | Name | When it's sent |
| ---- | ---- | -------------- |
| 200 | OK | Successful GET or PATCH |
| 201 | Created | Successful POST |
| 204 | No Content | Successful DELETE |
| 301 | Moved Permanently | URL has permanently changed |
| 400 | Bad Request | Request is malformed or missing required fields |
| 401 | Unauthorised | You're not logged in / no valid token |
| 403 | Forbidden | You're logged in but don't have permission |
| 404 | Not Found | The resource doesn't exist |
| 409 | Conflict | Duplicate resource (e.g. email already exists) |
| 422 | Unprocessable Entity | Validation failed |
| 429 | Too Many Requests | You've hit the rate limit |
| 500 | Internal Server Error | Something on the server crashed |

## JSON

JSON (JavaScript Object Notation) is the universal format for API data. It's a text format that represents objects, arrays, and values.

```json
{
  "id": 42,
  "name": "Alice Smith",
  "email": "alice@example.com",
  "age": null,
  "active": true,
  "tags": ["admin", "verified"],
  "address": {
    "city": "London",
    "country": "UK"
  }
}
```

JSON rules:
- Strings use double quotes (not single)
- No trailing commas
- No comments
- Numbers, booleans (`true`/`false`), and `null` have no quotes

In JavaScript/TypeScript:

```ts
// Parse JSON string → object
const user = JSON.parse(responseText);

// Convert object → JSON string
const body = JSON.stringify({ name: 'Alice', email: 'alice@example.com' });
```

## Authentication

Most APIs require authentication — proof that you're allowed to make this request. There are several patterns:

### API Keys

The simplest method. You're given a secret key when you register, and you send it with every request.

```
GET /data HTTP/1.1
X-API-Key: sk_live_abc123def456
```

Or in the URL (less secure): `GET /data?api_key=sk_live_abc123`

### Bearer Tokens (JWT)

After logging in, the server gives you a token. You send it in the `Authorization` header.

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOiI0MiJ9.abc
```

### Basic Auth

Base64-encoded username:password. Only acceptable over HTTPS.

```
Authorization: Basic YWxpY2U6cGFzc3dvcmQ=
```

### OAuth 2.0

A delegation protocol — lets users grant your app access to their data on another service (e.g., "Login with Google"). The user never gives you their Google password. Instead, Google gives your app a token that proves the user consented.

## Calling APIs from JavaScript

### Fetch API

The built-in browser API for making HTTP requests:

```ts
// GET request — read data
async function getUser(id: number) {
  const response = await fetch(`https://api.example.com/users/${id}`, {
    headers: {
      'Authorization': `Bearer ${token}`,
      'Accept': 'application/json',
    },
  });

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}: ${response.statusText}`);
  }

  const user = await response.json();
  return user;
}

// POST request — send data to create something
async function createUser(name: string, email: string) {
  const response = await fetch('https://api.example.com/users', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`,
    },
    body: JSON.stringify({ name, email }),
  });

  if (!response.ok) {
    const error = await response.json();
    throw new Error(error.message ?? 'Failed to create user');
  }

  return response.json();
}

// DELETE request
async function deleteUser(id: number) {
  const response = await fetch(`https://api.example.com/users/${id}`, {
    method: 'DELETE',
    headers: { 'Authorization': `Bearer ${token}` },
  });

  if (!response.ok) throw new Error(`Failed to delete user: ${response.status}`);
  // 204 No Content — no body to parse
}
```

### Handling errors properly

The `fetch` API only throws on network failures (no internet, DNS failure). A 404 or 500 response does **not** throw — you must check `response.ok` yourself.

```ts
async function safeGet<T>(url: string): Promise<T> {
  const response = await fetch(url, {
    headers: { Authorization: `Bearer ${getToken()}` },
  });

  if (response.status === 401) {
    logout(); // token expired
    throw new Error('Session expired');
  }

  if (response.status === 404) {
    throw new Error('Not found');
  }

  if (!response.ok) {
    const body = await response.text();
    throw new Error(`Request failed (${response.status}): ${body}`);
  }

  return response.json() as Promise<T>;
}
```

## API Design Principles

If you're building an API (not just consuming one), follow these conventions:

### Naming

```
✓ Nouns, plural:    /users, /products, /orders
✗ Verbs:            /getUsers, /createProduct, /deleteOrder

✓ Lowercase with hyphens: /blog-posts, /user-accounts
✗ CamelCase:              /blogPosts, /userAccounts
```

### Versioning

As your API evolves, breaking changes will happen. Version your API so existing clients don't break:

```
/api/v1/users    ← clients using the old API still work
/api/v2/users    ← new clients use the new API
```

### Pagination

Never return unlimited results. Use pagination:

```json
{
  "data": [...],
  "pagination": {
    "page": 2,
    "perPage": 20,
    "total": 347,
    "totalPages": 18,
    "nextPage": 3,
    "prevPage": 1
  }
}
```

```
GET /users?page=2&per_page=20
GET /users?cursor=eyJpZCI6NDJ9  ← cursor-based (better for large datasets)
```

### Consistent error responses

Always return errors in the same shape:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": [
      { "field": "email", "message": "Invalid email format" },
      { "field": "age", "message": "Must be a positive integer" }
    ]
  }
}
```

## Testing APIs

Before writing code that calls an API, test it directly:

### Thunder Client (VS Code extension)

A lightweight API client built into VS Code. Create requests, set headers, send bodies, and inspect responses — all without leaving the editor.

### Postman / Insomnia

Dedicated GUI apps for API testing. Good for saving collections of requests and sharing them with teams.

### curl (terminal)

The classic command-line tool for HTTP requests:

```bash
# GET request
curl https://api.example.com/users

# With headers
curl -H "Authorization: Bearer mytoken" https://api.example.com/users/42

# POST with JSON body
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice", "email": "alice@example.com"}'

# Pretty-print JSON response
curl https://api.example.com/users | python -m json.tool
```

## Rate Limiting

Most public APIs limit how many requests you can make in a given time window. If you exceed the limit, you'll get a `429 Too Many Requests` response.

The response typically includes headers telling you about the limits:

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 23
X-RateLimit-Reset: 1705323600
Retry-After: 60
```

In your code, handle 429 responses by waiting before retrying:

```ts
async function fetchWithRetry(url: string, retries = 3): Promise<Response> {
  const response = await fetch(url);

  if (response.status === 429 && retries > 0) {
    const retryAfter = parseInt(response.headers.get('Retry-After') ?? '1') * 1000;
    await new Promise(resolve => setTimeout(resolve, retryAfter));
    return fetchWithRetry(url, retries - 1);
  }

  return response;
}
```

## GraphQL

GraphQL is an alternative to REST developed by Facebook. Instead of many endpoints for different resources, GraphQL has a **single endpoint** and lets clients specify exactly what data they need.

```graphql
# REST: multiple requests needed:
# GET /users/42
# GET /users/42/posts
# GET /users/42/followers

# GraphQL: one request, get exactly what you need:
query GetUserWithPosts {
  user(id: "42") {
    name
    email
    posts(limit: 5) {
      title
      publishedAt
    }
    followers {
      name
    }
  }
}
```

The response only contains exactly what you asked for — no more, no less. GraphQL is powerful for complex data needs, but REST is simpler and more common for most APIs.

## WebSockets

HTTP is **request-response**: you ask, the server answers, the connection closes. For real-time applications where the server needs to push updates to the client (chat apps, live dashboards, multiplayer games), use **WebSockets**.

```ts
// Open a persistent connection
const socket = new WebSocket('wss://api.example.com/live');

socket.addEventListener('open', () => {
  console.log('Connected');
  socket.send(JSON.stringify({ type: 'subscribe', channel: 'prices' }));
});

socket.addEventListener('message', (event) => {
  const data = JSON.parse(event.data);
  updatePriceDisplay(data);
});

socket.addEventListener('close', () => {
  console.log('Disconnected — attempting to reconnect...');
  setTimeout(reconnect, 1000);
});
```
