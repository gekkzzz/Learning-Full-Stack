## Reconciliation and the Fibre Architecture

When state or props change, React needs to update the DOM. It does this efficiently through a process called **reconciliation**.

React maintains a **virtual DOM** — a lightweight JavaScript tree that mirrors the real DOM. When a re-render is triggered, React builds a new virtual DOM tree and **diffs** it against the previous one. Only the parts that changed are applied to the real DOM.

The modern React architecture is called **Fibre**. It represents the component tree as a linked list of "fibre" nodes, allowing React to:
- Pause, resume, and discard rendering work
- Assign priorities to different updates
- Split rendering across multiple frames (Concurrent Mode)

### What triggers a re-render?

- `setState` / `dispatch` called
- A parent re-renders (children re-render by default)
- Context value changes
- A hook's dependency changes

React re-renders are **not** DOM updates. Re-rendering just means calling the function component again. React then decides whether the DOM actually needs to change.

## Concurrent Features

React 18 introduced Concurrent Mode — the ability to prepare multiple versions of the UI at the same time and interrupt low-priority renders.

### `useTransition`

Mark a state update as non-urgent. React can interrupt it to handle more important updates (like user input).

```jsx
import { useTransition, useState } from 'react';

function SearchPage() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  function handleSearch(e) {
    setQuery(e.target.value); // urgent — update input immediately

    startTransition(() => {
      // Non-urgent — can be interrupted if user types again
      setResults(expensiveFilter(allData, e.target.value));
    });
  }

  return (
    <>
      <input value={query} onChange={handleSearch} />
      {isPending ? <Spinner /> : <ResultList items={results} />}
    </>
  );
}
```

### `useDeferredValue`

Defer a value to a lower-priority update — similar to `useTransition` but for values you don't control the setter for.

```jsx
function SearchResults({ query }) {
  const deferredQuery = useDeferredValue(query);
  const isStale = deferredQuery !== query;

  return (
    <div style={{ opacity: isStale ? 0.5 : 1 }}>
      <SlowList query={deferredQuery} />
    </div>
  );
}
```

## Suspense

Suspense lets components declare loading states declaratively. Instead of checking `isLoading` everywhere, you wrap the component in `<Suspense>` and provide a fallback.

```jsx
import { Suspense, lazy } from 'react';

// Code splitting — the component is loaded only when needed
const Dashboard = lazy(() => import('./Dashboard'));
const Chart = lazy(() => import('./Chart'));

function App() {
  return (
    <Suspense fallback={<PageSkeleton />}>
      <Dashboard />
    </Suspense>
  );
}

function Dashboard() {
  return (
    <div>
      <Suspense fallback={<ChartSkeleton />}>
        <Chart /> {/* loaded separately */}
      </Suspense>
    </div>
  );
}
```

### Suspense with data fetching

With TanStack Query and React 18:

```jsx
const { data } = useSuspenseQuery({
  queryKey: ['user', id],
  queryFn: () => fetchUser(id),
});
// No need to check isLoading — Suspense handles it above
```

## Error Boundaries

Error boundaries catch JavaScript errors anywhere in a child component tree, display a fallback UI, and log the error. They must be class components.

```jsx
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    logErrorToService(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? <h2>Something went wrong.</h2>;
    }
    return this.props.children;
  }
}

// Usage
<ErrorBoundary fallback={<ErrorPage />}>
  <FeatureSection />
</ErrorBoundary>
```

Libraries like `react-error-boundary` provide a hooks-friendly wrapper.

## Advanced State Patterns

### State machines with XState

For complex UI with many states and transitions, a state machine is more reliable than nested booleans:

```jsx
import { createMachine, assign } from 'xstate';
import { useMachine } from '@xstate/react';

const loginMachine = createMachine({
  initial: 'idle',
  states: {
    idle:    { on: { SUBMIT: 'loading' } },
    loading: { on: { SUCCESS: 'success', ERROR: 'failure' } },
    success: { type: 'final' },
    failure: { on: { RETRY: 'idle' } },
  },
});

function LoginForm() {
  const [state, send] = useMachine(loginMachine);

  async function handleSubmit(e) {
    e.preventDefault();
    send({ type: 'SUBMIT' });
    try {
      await loginUser();
      send({ type: 'SUCCESS' });
    } catch {
      send({ type: 'ERROR' });
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      {state.matches('failure') && <p>Login failed. <button onClick={() => send('RETRY')}>Retry</button></p>}
      <button disabled={state.matches('loading')}>Log in</button>
    </form>
  );
}
```

### Zustand (lightweight global state)

```jsx
import { create } from 'zustand';

const useStore = create((set) => ({
  user: null,
  setUser: (user) => set({ user }),
  logout: () => set({ user: null }),
}));

function Header() {
  const { user, logout } = useStore();
  return user ? <button onClick={logout}>{user.name}</button> : <LoginLink />;
}
```

## Render Props and Compound Components

### Render props

Pass a function as a prop to share logic while letting the parent control rendering:

```jsx
function MouseTracker({ render }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  return (
    <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
      {render(pos)}
    </div>
  );
}

<MouseTracker render={({ x, y }) => <p>Mouse at {x}, {y}</p>} />
```

### Compound components

Components that work together and share implicit state via context:

```jsx
const TabContext = createContext(null);

function Tabs({ children, defaultTab }) {
  const [active, setActive] = useState(defaultTab);
  return <TabContext.Provider value={{ active, setActive }}>{children}</TabContext.Provider>;
}

function Tab({ id, children }) {
  const { active, setActive } = useContext(TabContext);
  return (
    <button className={active === id ? 'active' : ''} onClick={() => setActive(id)}>
      {children}
    </button>
  );
}

function TabPanel({ id, children }) {
  const { active } = useContext(TabContext);
  return active === id ? <div>{children}</div> : null;
}

Tabs.Tab = Tab;
Tabs.Panel = TabPanel;

// Usage
<Tabs defaultTab="overview">
  <Tabs.Tab id="overview">Overview</Tabs.Tab>
  <Tabs.Tab id="details">Details</Tabs.Tab>
  <Tabs.Panel id="overview"><Overview /></Tabs.Panel>
  <Tabs.Panel id="details"><Details /></Tabs.Panel>
</Tabs>
```

### Identifying unnecessary re-renders

```jsx
// Why did this render? — add console to component
function UserCard({ user }) {
  console.count(`UserCard render: ${user.id}`);
  // ...
}

// Use React.memo with a custom comparator
const UserCard = React.memo(UserCard, (prev, next) => {
  return prev.user.id === next.user.id && prev.user.name === next.user.name;
});
```

## Server Components (RSC)

React Server Components render on the server and send HTML — or a compact serialised format — to the client. They have zero JavaScript bundle impact and can access databases and file systems directly.

```jsx
// app/users/page.tsx — Server Component (Next.js App Router)
// No 'use client' directive — runs only on the server
async function UsersPage() {
  const users = await db.user.findMany(); // direct database query, no API needed

  return (
    <ul>
      {users.map(u => (
        <li key={u.id}>
          {u.name}
          <DeleteButton userId={u.id} /> {/* Client Component for interactivity */}
        </li>
      ))}
    </ul>
  );
}
```

```jsx
// components/DeleteButton.tsx
'use client'; // marks this as a Client Component

function DeleteButton({ userId }) {
  const [deleting, setDeleting] = useState(false);
  // ... client-side interactivity
}
```

### Server vs Client Components

| | Server Component | Client Component |
| - | ---------------- | ---------------- |
| Runs on | Server only | Server (SSR) + Client |
| `useState`, `useEffect` | No | Yes |
| Event handlers | No | Yes |
| Access to DB, filesystem | Yes | No |
| Bundle size impact | None | Yes |
| Default in Next.js App Router | Yes | No (needs `'use client'`) |
