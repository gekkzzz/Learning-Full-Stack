# DevOps and AWS

---

## What Is DevOps?

DevOps is the practice of combining software **development** (Dev) and IT **operations** (Ops) into a unified workflow. The goal is to ship code faster and more reliably by automating the path from a developer's laptop to production servers.

In practical terms, DevOps means:
- Automating builds, tests, and deployments (CI/CD pipelines)
- Running applications in containers so they work the same everywhere
- Provisioning and managing cloud infrastructure
- Monitoring applications and responding to failures

As a full-stack developer you don't need to be a DevOps specialist, but you do need to understand this layer — because you're the one who writes the code that has to be deployed, scaled, and monitored.

## Docker — Containers

### Why Docker?

The classic developer problem: *"It works on my machine."* Docker solves this by packaging your application and all its dependencies (runtime, libraries, config) into a **container** — a lightweight, isolated unit that runs identically everywhere: your laptop, a colleague's machine, a CI server, and a production cloud server.

Containers are not virtual machines. They share the host OS kernel, making them much lighter and faster to start than VMs.

### Key concepts

| Term | What it is |
| ---- | ---------- |
| **Image** | A read-only blueprint for a container — built from a `Dockerfile` |
| **Container** | A running instance of an image |
| **Dockerfile** | Instructions for building an image |
| **Registry** | A place to store and share images (Docker Hub, AWS ECR, GitHub Container Registry) |
| **docker-compose** | A tool for running multiple containers together |

### Dockerfile

```dockerfile
# Start from an official Node.js base image
FROM node:20-alpine

# Set working directory inside the container
WORKDIR /app

# Copy package files first (so npm install is cached if source doesn't change)
COPY package*.json ./

# Install dependencies
RUN npm ci --only=production

# Copy application source
COPY . .

# Build the app (for Next.js / TypeScript projects)
RUN npm run build

# Expose the port the app listens on
EXPOSE 3000

# Start the server
CMD ["npm", "start"]
```

```dockerfile
# Python / FastAPI example
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Common Docker commands

```bash
# Build an image from the Dockerfile in the current directory
docker build -t my-app:latest .

# Run a container
docker run -p 3000:3000 my-app:latest
# -p host_port:container_port

# Run in detached mode (background)
docker run -d -p 3000:3000 --name my-container my-app:latest

# List running containers
docker ps

# Stop and remove
docker stop my-container
docker rm my-container

# View logs
docker logs my-container -f  # -f to follow (like tail -f)

# Shell inside a running container
docker exec -it my-container sh
```

### Docker Compose

For local development with multiple services (app + database + cache):

```yaml
# docker-compose.yml
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://postgres:password@db:5432/mydb
      REDIS_URL: redis://cache:6379
    depends_on:
      - db
      - cache
    volumes:
      - .:/app          # mount source code for hot reload
      - /app/node_modules  # don't overwrite container's node_modules

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: password
      POSTGRES_DB: mydb
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  postgres_data:
```

```bash
docker-compose up --build   # build and start all services
docker-compose up -d        # start in background
docker-compose down         # stop and remove containers
docker-compose logs -f app  # follow logs for the app service
```

## CI/CD — Automated Pipelines

**CI (Continuous Integration)** automatically runs tests every time you push code. If tests fail, the push is flagged and blocked from merging.

**CD (Continuous Deployment/Delivery)** automatically deploys passing code to staging or production.

### GitHub Actions

GitHub Actions is a CI/CD platform built into GitHub. You define workflows as YAML files in `.github/workflows/`.

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run type check
        run: npm run type-check

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build
```

```yaml
# .github/workflows/deploy.yml — deploy to production on merge to main
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    needs: test  # only deploy if tests passed

    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id:     ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: eu-west-1

      - name: Login to Amazon ECR
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build and push Docker image
        run: |
          docker build -t $ECR_REGISTRY/my-app:$GITHUB_SHA .
          docker push $ECR_REGISTRY/my-app:$GITHUB_SHA
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}

      - name: Deploy to ECS
        run: aws ecs update-service --cluster my-cluster --service my-service --force-new-deployment
```

## AWS — Amazon Web Services

AWS is the largest cloud provider. It offers over 200 services — from virtual servers to managed databases, object storage, CDNs, and AI services. You can build and run almost any application on AWS.

### Core services every web developer should know

| Service | What it does |
| ------- | ------------ |
| **EC2** | Virtual servers (Linux/Windows VMs) |
| **ECS / Fargate** | Run Docker containers without managing servers |
| **S3** | Object storage — files, images, backups |
| **RDS** | Managed SQL databases (PostgreSQL, MySQL) |
| **ElastiCache** | Managed Redis/Memcached |
| **CloudFront** | CDN — serves content from edge locations worldwide |
| **Route 53** | DNS — maps domain names to IP addresses |
| **ECR** | Docker image registry |
| **Secrets Manager** | Store and rotate secrets (API keys, DB passwords) |
| **CloudWatch** | Logs, metrics, and alerts |
| **IAM** | Identity and access management — who can do what |

## AWS S3 — File Storage

S3 (Simple Storage Service) stores any file — images, videos, backups, build artefacts — at virtually unlimited scale.

```bash
pip install boto3  # Python AWS SDK
npm install @aws-sdk/client-s3  # Node.js
```

```python
# Python — upload a file to S3
import boto3

s3 = boto3.client('s3',
    region_name='eu-west-1',
    aws_access_key_id=os.environ['AWS_ACCESS_KEY_ID'],
    aws_secret_access_key=os.environ['AWS_SECRET_ACCESS_KEY'],
)

# Upload
s3.upload_file('local-file.jpg', 'my-bucket', 'uploads/photo.jpg')

# Download
s3.download_file('my-bucket', 'uploads/photo.jpg', 'downloaded.jpg')

# Generate a pre-signed URL (lets a user download without credentials, expires)
url = s3.generate_presigned_url(
    'get_object',
    Params={'Bucket': 'my-bucket', 'Key': 'uploads/photo.jpg'},
    ExpiresIn=3600  # 1 hour
)
```

```typescript
// TypeScript — upload to S3 from a Next.js API route
import { S3Client, PutObjectCommand, GetObjectCommand } from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

const s3 = new S3Client({ region: 'eu-west-1' });

// Upload file
const command = new PutObjectCommand({
  Bucket: process.env.S3_BUCKET!,
  Key: `uploads/${Date.now()}-${filename}`,
  Body: fileBuffer,
  ContentType: 'image/jpeg',
});
await s3.send(command);

// Pre-signed URL for private files
const getCommand = new GetObjectCommand({ Bucket: process.env.S3_BUCKET!, Key: key });
const signedUrl = await getSignedUrl(s3, getCommand, { expiresIn: 3600 });
```

## AWS RDS — Managed PostgreSQL

RDS handles backups, patching, and failover for your database so you don't have to manage a server.

```
1. Create a PostgreSQL instance in RDS (Free tier: db.t3.micro)
2. Choose: Multi-AZ? (yes for production, no for dev)
3. Set master username and password
4. Note the endpoint: mydb.xyz.eu-west-1.rds.amazonaws.com
5. Set security group to allow inbound on port 5432 from your app's IP / security group
```

Connection string:

```
postgresql://username:password@mydb.xyz.eu-west-1.rds.amazonaws.com:5432/mydb
```

Store this in AWS Secrets Manager, not in your code.

## ECS and Fargate — Running Docker in Production

**ECS** (Elastic Container Service) is AWS's container orchestration platform. **Fargate** is a launch type for ECS where AWS manages the underlying servers — you only think about containers.

Conceptually:

```
Cluster → Service → Task → Container
```

- **Cluster** — logical grouping of resources
- **Service** — keeps N copies of your Task running; replaces failed containers; handles rolling deploys
- **Task Definition** — specifies which Docker image, how much CPU/RAM, environment variables, ports
- **Container** — the running Docker container

A typical deployment flow:

```
1. Build Docker image
2. Push to ECR (AWS's Docker registry)
3. Update Task Definition with new image tag
4. ECS Service deploys new containers, drains old ones
```

## Environment Variables and Secrets

Never put secrets (API keys, database passwords) in your code or Docker images. Use environment variables, injected at runtime from a secrets manager.

```bash
# AWS Secrets Manager — store a secret
aws secretsmanager create-secret \
  --name prod/myapp/database \
  --secret-string '{"password":"supersecret","url":"postgresql://..."}'

# Retrieve in Python
import boto3, json

client = boto3.client('secretsmanager', region_name='eu-west-1')
secret = client.get_secret_value(SecretId='prod/myapp/database')
db_config = json.loads(secret['SecretString'])
```

In ECS Task Definitions, you can reference Secrets Manager secrets directly — they are injected as environment variables when the container starts.

## CloudWatch — Logs and Monitoring

CloudWatch collects logs from your containers, metrics (CPU, memory, request count), and lets you set alarms.

```python
# Python — structured logging that CloudWatch can parse
import logging
import json

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def log(level, message, **extra):
    logger.log(level, json.dumps({'message': message, **extra}))

log(logging.INFO, 'User created', user_id=42, email='alice@example.com')
log(logging.ERROR, 'Payment failed', order_id=123, error=str(e))
```

Key CloudWatch concepts:

- **Log Group** — a collection of log streams (one per service)
- **Log Stream** — logs from one container instance
- **Metric Filter** — extract a metric from log data (e.g., count of ERROR logs)
- **Alarm** — trigger a notification or action when a metric crosses a threshold
- **Dashboard** — visualise metrics

## IAM — Who Can Do What

IAM (Identity and Access Management) controls access to AWS resources. The principle of least privilege: give each service only the permissions it actually needs.

```json
// IAM policy — allow reading from a specific S3 bucket only
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::my-bucket/uploads/*"
    }
  ]
}
```

Key IAM concepts:

- **User** — a person with long-term credentials (avoid for applications)
- **Role** — temporary credentials assumed by a service (use for EC2, ECS, Lambda)
- **Policy** — a JSON document that grants or denies permissions
- **Group** — a collection of users that share a policy

Best practices:
- Never use root account credentials
- Never put AWS keys in your code — use IAM roles for services running on AWS
- Enable MFA on all user accounts
- Rotate access keys regularly

## A Typical Production Architecture

```
Users
  │
  ▼
CloudFront (CDN)
  │
  ├─── Static assets (S3 bucket) ─────────── Next.js static files, images
  │
  └─── API requests
         │
         ▼
    Application Load Balancer
         │
         ▼
    ECS Fargate (Docker containers)
    ┌────────────┐  ┌────────────┐
    │  App Task  │  │  App Task  │  ← auto-scales with load
    └─────┬──────┘  └─────┬──────┘
          │               │
          ▼               ▼
    RDS PostgreSQL      ElastiCache Redis
    (primary + replica) (sessions, caching)

    ↑ All secrets from Secrets Manager
    ↑ All logs to CloudWatch
    ↑ Deployments triggered by GitHub Actions
```

## Getting Started — Recommended Learning Path

1. **Create a free AWS account** — many services have a free tier
2. **Deploy a static site to S3 + CloudFront** — learn S3, buckets, policies
3. **Launch an RDS PostgreSQL instance** — connect from your local machine
4. **Containerise your app with Docker** — write a Dockerfile, test locally
5. **Push to ECR and deploy to ECS Fargate** — the core production pattern
6. **Set up a GitHub Actions CI/CD pipeline** — automate deployments
7. **Add CloudWatch logging and alarms** — know when things break

```bash
# Install the AWS CLI
pip install awscli
# or on Mac: brew install awscli

# Configure credentials
aws configure
# Enter: AWS Access Key ID, Secret Access Key, region (eu-west-1), output format (json)

# Verify
aws sts get-caller-identity
aws s3 ls
```
